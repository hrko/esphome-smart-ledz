# SmartLEDZ Export YAML SPA Requirements

## Goal

Build a static SPA that reads SmartLEDZ Fit raw export JSON and generates ESPHome YAML for the `smart_ledz` external component.

## Required Output Sections

The generated ESPHome YAML must include exactly these top-level objects:

1. `external_components`
2. `ble_client`
3. `smart_ledz`
4. `light`

No additional top-level objects are generated by this tool.

## External Component Source

`external_components` must always point to:

- `url: https://github.com/hrko/esphome-smart-ledz.git`
- `ref: main`

## Input Format

Supported input is SmartLEDZ Fit raw export JSON (not normalized JSON).

The SPA reads fields such as:

- `meshName`, `meshPassword`
- `deviceArr`, `deviceAddrArr`, `deviceMac`
- `groupArr`, `group_p_Arr`

## BLE Client Selection

The UI must provide a device selector for `ble_client`.

- Default selected option: first parsed device.
- Option label format: `device_name (MAC)`.
- `ble_client` node settings are editable:
  - `id`
  - `mac_address`
  - `auto_connect`

## Secret Handling

`smart_ledz.mesh_name` and `smart_ledz.mesh_password` in generated YAML must be `!secret` references.

The SPA must also output a separate `secrets.yaml` block containing actual values:

- secret key for mesh name (editable key name)
- secret key for mesh password (editable key name)

## Light Defaults and Overrides

For each selected light entry:

- `name` default comes from backup data and is editable.
- `id` default is `ledz_light_<target_address>` and is editable.
- `target` is editable.

## Selection Scope

The UI must allow users to select conversion targets:

- individual devices
- groups

Only selected items are included in the generated `light` list.

## Group Target Rule

For groups, default `target` is computed as:

- `0x8000 | group_low_byte`

User can override this value in the UI.

## Configurability

The SPA must provide configurable settings:

- hub-level settings (`smart_ledz`)
- light-level settings (`light`)

At minimum:

- `smart_ledz`: `id`, `ble_client_id`, `vendor_id`, `poll_interval`, `tx_interval`, `power_on_settle`, extra YAML lines
- `light`: `id`, `name`, `target`, `device_type`, `ct_duv`, `ignore_transition`, extra YAML lines

## Device Type Default Mapping

Initial `device_type` should be inferred from type code:

- `1, 53, 6, 8, 7` -> `dimmable`
- `2, 54` -> `tunable`
- `52, 48, 55` -> `synca`
- unknown -> `dimmable`

## Output UX

The page must render output in code blocks and provide copy buttons for:

1. generated ESPHome YAML
2. generated `secrets.yaml`

## Validation

The SPA must validate and report errors for at least:

- invalid JSON
- missing required mesh credentials
- no selected light targets
- invalid or duplicate IDs
- invalid target format

## Implementation Constraints

- Implement inside submodule repository (`esphome-smart-ledz`).
- Build as a static SPA (`index.html`, `styles.css`, `app.js`) with no framework dependency.
